The day begins with routines like waking up and preparing for the day, followed by a short morning care routine to establish a productive start and so on so on.

There are 2 types of tasks: fixed time and flexible time, fixed time tasks have their strict starting time and needed to be done exactly at required time. Flexible time tasks are mostly builded around them.

Each task has its duration - in my case tasks with more time are higher priority, since it means they are more serious and require high amount of energy.

Each task has dependency feature - it means that this task can not run until every task it is dependent on will be completed, by that Go to Sleep has all tasks as dependencies, since I dont want to end day without completing all tasks, or for example Medicines task - by prescription I have to eat before taking these medications.

class MaxHeap:
    """
    Max heap data structure to manage tasks based on their priority.
    The task with the highest priority will always be at the root of the heap.
    """

    def __init__(self):
        self.heap = []  # Empty heap to store tasks

    def heappush(self, task):
        """
        Adds a task to the heap and maintains the heap property by bubbling the task up.
        Parameters:
        task (int): The priority value of the task (or the task itself if objects are used).
        """
        self.heap.append(task)  # Add the task to the end of the heap
        self._bubble_up(len(self.heap) - 1)  # Reorder the heap to ensure the task is placed correctly

    def heappop(self):
        """
        Removes and returns the task with the highest priority (root of the heap).

        Returns:
            int: The task with the highest priority.

        Raises:
            IndexError: If the heap is empty and there's no task to pop.
        """
        if not self.heap:
            raise IndexError("Heap is empty")  # Error if the heap is empty

        if len(self.heap) == 1:
            return self.heap.pop()

        max_task = self.heap[0]
        self.heap[0] = self.heap.pop()  # Replace the root with the last task in the heap
        self._heapify(0)
        return max_task

    def _bubble_up(self, index):
        while index > 0:
            parent_index = self.parent(index)
            if self.heap[index] > self.heap[parent_index]:  # Ensuring max-heap property
                self.heap[index], self.heap[parent_index] = self.heap[parent_index], self.heap[index]
                index = parent_index
            else:
                break

    def _heapify(self, i):
        largest = i
        left_child = self.left(i)
        right_child = self.right(i)

        if left_child < len(self.heap) and self.heap[left_child] > self.heap[largest]:
            largest = left_child
        if right_child < len(self.heap) and self.heap[right_child] > self.heap[largest]:
            largest = right_child
        if largest != i:
            self.heap[i], self.heap[largest] = self.heap[largest], self.heap[i]
            self._heapify(largest)

    def left(self, i):
        """
        Returns the index of the left child of the node at index i.

        Parameters:
        i (int): The index of the current node.

        Returns:
        int: The index of the left child.
        """
        return 2 * i + 1

    def right(self, i):
        """
        Returns the index of the right child of the node at index i.

        Parameters:
        i (int): The index of the current node.

        Returns:
        int: The index of the right child.
        """
        return 2 * i + 2

    def parent(self, i):
        """
        Returns the index of the parent of the node at index i.

        Parameters:
        i (int): The index of the current node.

        Returns:
        int: The index of the parent.
        """
        return (i - 1) // 2

    def __len__(self):
        return len(self.heap)

    def reset_heap(self, tasks):
        """
        Refills the heap with tasks to refresh its structure.
        Parameters:
        tasks (list): List of tasks or priorities to refill the heap.
        """
        self.heap = []
        for task in tasks:
            self.heappush(task)

class Task:
    def __init__(self, id, description, duration, dependencies, is_fixed=False, start_time=None, status="N", importance="medium"):
        self.id = id
        self.description = description 
        self.duration = duration #duration of task
        self.dependencies = dependencies
        self.is_fixed = is_fixed #if task has fixed time
        self.start_time = start_time #it will have fixed starting time
        self.priority = 0 
        self.status = status 
        self.importance = importance  #importance level "high", "medium", or "low" 

    def calculate_priority(self, current_time=None):
        """
        Calculate priority for tasks with the following logic:
        Fixed tasks always have 100 utility points.
        Flexible tasks have a base of 100 and decline based on penalties:
        - Dependency count
        - Duration (longer tasks are penalized less, since they are more important)
        - Importance level (higher importance reduces penalties)
        """
        if current_time is None:
            current_time = 9 * 60  #default to 9:00 AM

        if self.is_fixed:
            #fixed tasks always have a consistent utility score of 100, because they are fixed
            self.priority = 200
        else:
            #base priority for flexible tasks starts from 100 as well
            base = 100

            #dependency penalty
            dependency_penalty = len(self.dependencies) * 10  # 10 points per dependency penalized

            #duration penalty (shorter tasks penalized more to incentivize longer ones)
            duration_penalty = max(0, 40 - (self.duration // 30) * 10)

            #importance bonus (higher importance decreases penalty, and adds for utility)
            importance_bonus = {"high": 30, "medium": 15, "low": 0}[self.importance]

            #final priority calculation for flexible tasks
            #self.priority = base - dependency_penalty - duration_penalty + importance_bonus
            self.priority = 100 + (importance_bonus * 2) - dependency_penalty - duration_penalty

            #ensure priority doesn't exceed base or drop below 0
            self.priority = round(max(0, min(base, self.priority)))
        

    def __lt__(self, other):
        return self.priority > other.priority  #reverse for max-heap


class TaskScheduler:
    NOT_STARTED = 'N'
    IN_PRIORITY_QUEUE = 'I'
    COMPLETED = 'C'
    
    def __init__(self, tasks):
        self.tasks = tasks
        self.priority_queue = MaxHeap()
        
    def remove_dependency(self, id):
        for t in self.tasks:
            if t.id != id and id in t.dependencies:
                t.dependencies.remove(id)
    
    def get_tasks_ready(self, current_time):
        #first, add fixed tasks that are due now
        for task in self.tasks:
            if (task.status == self.NOT_STARTED and 
                task.is_fixed and 
                task.start_time == current_time and 
                not task.dependencies):
                task.calculate_priority()
                task.status = self.IN_PRIORITY_QUEUE
                self.priority_queue.heappush(task)
        
        #then add flexible tasks building around fixed tasks
        if len(self.priority_queue) == 0:
            for task in self.tasks:
                if (task.status == self.NOT_STARTED and 
                    not task.dependencies and 
                    (not task.is_fixed or current_time >= task.start_time)):
                    task.calculate_priority()
                    task.status = self.IN_PRIORITY_QUEUE
                    self.priority_queue.heappush(task)
    
    def get_next_fixed_task_time(self, current_time):
        next_time = None
        for task in self.tasks:
            if (task.status == self.NOT_STARTED and 
                task.is_fixed and 
                task.start_time > current_time):
                if next_time is None or task.start_time < next_time:
                    next_time = task.start_time
        return next_time
    
    def update_priorities(self):
        for task in self.tasks:
            if task.status == self.NOT_STARTED:
                task.calculate_priority()
        #rebuild the heap only for tasks in the queue
        self.priority_queue.reset_heap([t for t in self.tasks if t.status == self.IN_PRIORITY_QUEUE])
    
    def check_unscheduled_tasks(self):
        return any(task.status == self.NOT_STARTED for task in self.tasks)
    
    def format_time(self, time):
        return f"{time // 60}h{time % 60:02d}"

    def run_task_scheduler(self, starting_time, completed_task_order=None, suppress_output=False):
        current_time = starting_time
        total_utils = 0

        if not suppress_output:
            print("Running a priority-based scheduler:\n")

        while self.check_unscheduled_tasks():
            #recalculate priorities dynamically for all tasks
            self.update_priorities()  #dynamic recalculation
            self.get_tasks_ready(current_time)
            
            if len(self.priority_queue) > 0:
                task = self.priority_queue.heappop()
                if not suppress_output:
                    print(f"t={self.format_time(current_time)}")
                    task_type = " (Fixed Task)" if task.is_fixed else ""
                    print(f"\tstarted '{task.description}' for {task.duration} mins{task_type}, utils = {task.priority}.")
                current_time += task.duration
                total_utils += task.priority
                self.remove_dependency(task.id)
                task.status = self.COMPLETED
                #recalculate priorities after task completion
                self.update_priorities()
                if completed_task_order is not None:
                    completed_task_order.append(task.id)
                if not suppress_output:
                    print(f"\t t={self.format_time(current_time)}, task completed!")
            else:
                #jump to next fixed task time if available
                next_time = self.get_next_fixed_task_time(current_time)
                if next_time is not None:
                    current_time = next_time
                else:
                    #no more fixed tasks, increment time by 1 minute
                    current_time += 1

        total_time = current_time - starting_time
        if not suppress_output:
            print(f"\n Completed all planned tasks in {total_time // 60}h{total_time % 60:02d}min!")
            print(f"Total utility points (utils) accumulated: {total_utils}")

The scheduler implementation shows significant strengths in its ability to manage both fixed and flexible tasks efficiently. Fixed tasks, such as my classes or time-sensitive fixed work on project, are scheduled at their designated times, ensuring that critical commitments are met without any conflicts. At same time, flexible tasks are dynamically arranged around these fixed activities, filling between periods and optimizing the overall schedule. This capability ensures that time is utilized very effectively, balancing rigidity with adaptability.

Main advantage of the scheduler is its dynamic prioritization mechanism. Each tasks priority is calculated based on dependencies, importance, and duration metrics. Tasks with fewer or no dependencies are prioritized, allowing prerequisite tasks to be addressed in its correct order. Importance levels are also included into the priority calculations, giving higher weight to tasks marked as high importance to ensure they are completed early in the schedule. Additionally, tasks with longer durations are incentivized, as for my understanding of higher importance task is acknowledging their potential demand for greater energy and focus. This combination of parameters ensures that the scheduler adapts intelligently as tasks are completed, providing a balanced approach to task management.

One remark is that fixed time tasks always has 100 points of utils, and even if its higher than task beforehand by util points, we should remember that task is fixed, it cant be pulled before its designated time, so my algorithm builds flexible time tasks after and around fixed time tasks.

The utility formula balances dependencies, duration, importance, and fixed scheduling constraints to prioritize tasks effectively:

Base Utility: Every task starts with 100 points.

Dependency Penalty: Each dependency reduces utility by 10 points.

Duration Bonus: Longer tasks are favored, with a 10-point bonus for every 30-minute block (up to 40 points).

Importance Bonus: High-importance tasks get 30 points, medium 15, and low 0.

Fixed Tasks: Always assigned 100 utility points to ensure scheduling at their designated times.

Flexible tasks:
Utility= 100 (Dependencies * 10)Duration Penalty + Importance Bonus

Fixed tasks:
Utility = 100

This dynamic formula ensures tasks are prioritized based on significance, dependencies, and time sensitivity, optimizing daily productivity.

Lastly, the scheduler excels in minimizing idle time and ensuring an optimal task completion order. By using a priority queue, it transitions from one task to the next, ensuring no gaps between activities unless required. The integration of fixed and flexible tasks is handled smoothly, avoiding scheduling conflicts and providing a streamlined workflow.

Disadvantages of the Scheduler:

But of course there are some disadvantages of this implementation, for example the utility formula might oversimplify real-life task complexities, ignoring factors like task urgency or user(me) energy levels. Also the algorithm might still generate minor idle gaps if no flexible tasks align with upcoming fixed tasks based on duration. Instead of simply assigning 100 utils I could implement better system.

B. Failures and Patches

Initially, overlapping fixed tasks created scheduling conflicts, causing one task to be missed. This was mitigated by ensuring fixed tasks are strictly ordered based on their start_time and flagged as high-priority to avoid overlaps.

Flexible tasks without dependencies were not always being scheduled efficiently. This issue was addressed by dynamically recalculating their priorities, giving higher weight to tasks with greater importance and duration.

Assigning priorities purely based on duration and dependencies resulted in some critical tasks being scheduled late. A patch was introduced by adding the importance feature, which allowed tasks to be weighted according to their significance (high, medium, or low). This really made sure that essential tasks like healthcare routines like taking medicines were given appropriate precedence, even among flexible tasks.

These patches improved the schedulers efficiency and reliability in managing complex task schedules.

Complexity of MaxHeap implementation

Time Complexity

We will consider and analyze MaxHeap class implementations time complexity, breaking it down into individual methods like heappush, heappop, bubble_up (alternative name for increase_key), and others.

  1. heappush(task)
    Best Case: O(log n)- Inserting into the heap requires bubbling up to maintain the heap property. The height of the heap is proportional to log(n), where n is the number of elements.
    Worst Case: O(log n) - The maximum number of swaps occurs in a complete binary tree, still proportional to log(n).

  2. heappop()
    Best Case: O(log n) - Removing the root requires replacing it with the last element and performing a heapify operation, which traverses the height of the heap (log n).
    Worst Case: O(log n) - Similar to the best case; the number of swaps is proportional to log(n).

  3. Other Methods Used in class
    left(i), right(i), parent(i): O(1) - These methods involve simple arithmetic operations to calculate indexes.
    _bubble_up(index) or increase_key: O(log n) - This method traverses the height of the heap to maintain the heap property.
    _heapify(index): O(log n)- Similar to _bubble_up, this method also works on the height of the heap.

Space Complexity

  • Heap Storage: O(n)
    • The heap is stored in a list, and n is the total number of elements in the heap.

Runtime efficiency based on steps metrics

import matplotlib.pyplot as plt

"""
This short script evaluates the runtime efficiency of a custom MaxHeap implementation 
by measuring the number of steps required for `heappush` and `heappop` 
operations as the heap size increases.

Classes:
- MaxHeapWithSteps: Implements a max-heap with methods for insertion (`heappush`) 
  and removal (`heappop`), while tracking the number of steps taken for comparisons and swaps.

Process:
1. Measure steps for inserting elements into the heap (`heappush`).
2. Measure steps for removing elements from the heap (`heappop`).
3. Plot the number of steps for both operations against varying heap sizes.

Outputs:
- A plot visualizing the relationship between heap size and the number of operations 
  required for `heappush` and `heappop`, showing the logarithmic growth of steps 
  consistent with O(log n) complexity.
"""


class MaxHeapWithSteps:
    def __init__(self):
        self.heap = []
        self.steps = 0  #to count the number of steps

    def heappush(self, task):
        self.heap.append(task)
        self._bubble_up(len(self.heap) - 1)

    def heappop(self):
        if not self.heap:
            raise IndexError("Heap is empty")
        if len(self.heap) == 1:
            return self.heap.pop()
        max_task = self.heap[0]
        self.heap[0] = self.heap.pop()
        self._heapify(0)
        return max_task

    def _bubble_up(self, index):
        while index > 0:
            parent_index = self.parent(index)
            self.steps += 1  #increment step for comparison
            if self.heap[index] > self.heap[parent_index]:
                self.steps += 1  #increment step for swap
                self.heap[index], self.heap[parent_index] = self.heap[parent_index], self.heap[index]
                index = parent_index
            else:
                break

    def _heapify(self, i):
        largest = i
        left_child = self.left(i)
        right_child = self.right(i)

        if left_child < len(self.heap):
            self.steps += 1  #increment step for comparison
            if self.heap[left_child] > self.heap[largest]:
                largest = left_child

        if right_child < len(self.heap):
            self.steps += 1  #increment step for comparison
            if self.heap[right_child] > self.heap[largest]:
                largest = right_child

        if largest != i:
            self.steps += 1  #increment step for swap
            self.heap[i], self.heap[largest] = self.heap[largest], self.heap[i]
            self._heapify(largest)

    def left(self, i):
        return 2 * i + 1

    def right(self, i):
        return 2 * i + 2

    def parent(self, i):
        return (i - 1) // 2

    def reset_steps(self):
        self.steps = 0

    def get_steps(self):
        return self.steps

heap_sizes = [10, 50, 100, 500, 1000]  #different heap sizes
steps_push = []  #to store steps for push
steps_pop = []  #to store steps for pop

for size in heap_sizes:
    heap = MaxHeapWithSteps()
    data = list(range(size))  

    #measure steps for heappush
    heap.reset_steps()
    for val in data:
        heap.heappush(val)
    steps_push.append(heap.get_steps())

    #measure steps for heappop
    heap.reset_steps()
    while heap.heap:
        heap.heappop()
    steps_pop.append(heap.get_steps())

plt.figure(figsize=(10, 5))
plt.plot(heap_sizes, steps_push, label="heappush steps")
plt.plot(heap_sizes, steps_pop, label="heappop steps")
plt.xlabel("Heap Size")
plt.ylabel("Steps")
plt.title("Runtime Efficiency of MaxHeap")
plt.legend()
plt.grid()
plt.show()

Visualization of Efficiency:
The plot above compares the number of steps for heappush and heappop as the heap size increases. It highlights the logarithmic growth of steps, consistent with theoretical time complexity (O(log n)).

The number of steps for heappop appear slightly higher than heappush, because heappop involves additional operations like swapping the root with the last element before heapifying. Our plot effectively visualizes it.

Complexity of Task and TaskScheduler classes

Complexity Analysis for Task and TaskScheduler Classes

Time Complexity

1. Task Class:

  • calculate_priority(): O(1)
    Summary: The computation for priority is constant time as it doesnt depend on the number of tasks.

2. TaskScheduler Class:

  • __init__:
    Initializes tasks and priority queue. O(1) since this step involves only assignments.

  • remove_dependency(id): Iterates through all tasks to remove a specific dependency.
    Best Case: O(1) (if no tasks have the given dependency).
    Worst Case: O(n) (where n is the number of tasks).

  • get_tasks_ready(current_time): Iterates over tasks to identify those ready to execute.
    Best Case: O(n) (if no tasks are ready or all tasks are checked in one pass).
    Worst Case: O(n log n) (if priorities are recalculated and tasks are added to the heap).

  • get_next_fixed_task_time(current_time): Iterates through tasks to find the next fixed task.
    Best Case: O(1) (if the next fixed task is the first in the list).
    Worst Case: O(n).

  • update_priorities(): Iterates through tasks to recalculate priorities and rebuild the heap.
    Best Case: O(n).
    Worst Case: O(n log n) (heap rebuilding for all tasks in the priority queue).

  • run_task_scheduler(starting_time): While loop iterates over tasks until all are completed.
    Best Case: O(n log n) (minimal dependencies, each task added to the heap once).
    Worst Case: O(mlogk+n+klogk) (complex dependencies requiring multiple recalculations and additions to the heap) where:

Checking unscheduled tasks: O(n)

Updating priorities: O(n+klogk)

Running tasks: O(mlogk)

Space Complexity

1. Task Class:
Each task uses a fixed amount of memory for its fields.
Space Complexity: O(1) per task.

2. TaskScheduler Class:
Heap Storage: O(n) for all tasks in the priority queue.
Task List: O(n) to store all tasks.
Total Space Complexity: O(n).

import time
import matplotlib.pyplot as plt

"""
This short code evaluates the empirical time complexity of a custom `TaskScheduler` 
implementation by measuring the execution time and steps required for scheduling tasks 
as the number of tasks increases.

Function:
- `test_task_scheduler_complexity`: 
  1. Generates tasks with random dependencies for different input sizes.
  2. Adds fixed-time tasks to simulate real-world scheduling constraints.
  3. Measures the total steps and time required to schedule and execute all tasks.

Process:
1. Measure execution time for scheduling tasks with varying input sizes.
2. Collect the number of steps (operations) performed during task scheduling.
3. Visualize the relationship between task count and execution time.

Outputs:
- A plot showing the execution time vs. the number of tasks, illustrating how the scheduler 
  scales with increasing input sizes.
"""


# Function to test complexity
def test_task_scheduler_complexity():
    task_sizes = [10, 50, 100, 500, 1000, 1500, 2000, 3500]  #growing number of tasks
    steps_taken = []
    time_taken = []

    for size in task_sizes:
        #generate tasks with random dependencies
        tasks = [
            Task(id=i, description=f'Task {i}', duration=10, dependencies=[], importance="medium")
            for i in range(size)
        ]
        #add some fixed tasks
        if size >= 10:
            tasks[0].is_fixed = True
            tasks[0].start_time = 9 * 60
            tasks[1].is_fixed = True
            tasks[1].start_time = 10 * 60

        scheduler = TaskSchedulerWithSteps(tasks)
        scheduler.reset_steps()

        start_time = time.time()
        scheduler.run_task_scheduler(starting_time=9 * 60, suppress_output=True)
        end_time = time.time()

        steps_taken.append(scheduler.get_steps())
        time_taken.append(end_time - start_time)
        
        

    plt.figure(figsize=(10, 5))
    plt.plot(task_sizes, time_taken, label="Time Taken")
    plt.xlabel("Number of Tasks")
    plt.ylabel("Time (seconds)")
    plt.title("Time Taken vs Number of Tasks")
    plt.legend()
    plt.grid()
    plt.show()

Comparing theoretical analysis with empirical findings

This plot shows the time taken by the task scheduler with respect to the number of tasks. By observing the shape of the curve - we can see that the curve increases at a slower rate as the number of tasks grows, suggesting a logarithmic or sub-linear behavior initially, then flattens out, likely due to constant overhead or garbage collection at the higher task count.

  1. This empirical result aligns with the theoretical complexity of (O(n + klog k + mlog k)), where:

    • (n) is the number of tasks.
    • (k) is the size of the heap (priority queue).
    • (m) is the number of executed tasks.

  2. The flat portion might be caused by system optimizations or constraints such as caching or task dependencies becoming a limiting factor.

The plot suggests an average case close to (O(n* log k), where (k) is bounded by a smaller number (tasks added to the priority queue), making the growth manageable.

The empirical results align nicely with the theoretical analysis, especially for larger inputs where the scaling pattern closely matches the expected behavior. Deviations for smaller inputs are likely due to overheads such as Pythons garbage collection and measurement inaccuracies. These results confirm that the scheduler efficiently handles smaller task sets but becomes significantly slower for larger inputs due to the bottlenecks in run_task_scheduler, particularly during dependency resolution (we discussed in disadvantages) and heap operations. The findings highlight the need for optimization in handling complex dependencies and recalculations for scalability in larger datasets.

Future Improvements

The current version of the scheduler demonstrates strong task management capabilities. However, there is always room for self-feedback and reflection for future improvements in several areas to enhance its scalability, flexibility, and alignment with real-world scenarios:

  1. Dynamic Task Updates: The scheduler currently assumes a static list of tasks that you can only upload once. Incorporating a dynamic feature to allow real-time addition, modification, or removal of tasks during execution would make the scheduler more attractive to use for users. This could be particularly useful for adapting to unforeseen changes or emergencies in the schedule.

  2. Improved Priority Calculation: The existing utility function already balances dependencies, duration, and importance, but it could be improved to include additional factors:
    -Incorporating an urgency metric to prioritize tasks based on how close they are to a deadline.
    -Considering user energy patterns throughout the day, giving higher priority to demanding tasks during peak energy hours (which can be chosen by the user).

  3. Support for Parallel Tasks at the same time: The current implementation assumes tasks are executed only sequentially. Adding support for parallel processing of tasks (allowing overlapping tasks that do not conflict) would increase efficiency, especially for group projects or multitasking.

With these improvements, the scheduler could become a much more adaptive, efficient, and user-friendly tool for managing complex daily routines and schedules.
